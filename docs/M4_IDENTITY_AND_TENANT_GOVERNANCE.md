## 2. M3 regressions repaired before identity work

M4-A first repaired two defects found in the merged M3 collaboration boundary.

### 2.1 Exact task outcome mapping

Commit `d948ae6cb4f8ef38453b236f6e0b4ae5e2fd1963` replaced string-derived task outcome construction with an exact action-to-domain-outcome map:

- `approve` -> `APPROVED`;
- `reject` -> `REJECTED`;
- `resubmit` -> `RESUBMITTED`.

This prevents the reject path from producing the invalid value `REJECTD`. The interceptor clears stale thread-local outcome state before and after every request.

### 2.2 Rejected parent task terminates active collaboration

Commit `fdc53227818791e6702a6003edfc4debc1a76af4` made parent rejection terminate active add-sign collaboration evidence in the same transaction boundary:

1. the active collaboration and original task owner are read under the existing task/collaboration lock boundary;
2. the parent task projection and active task set are committed first;
3. the active collaboration policy and remaining pending participants are canceled with a terminal reason;
4. approval and other non-rejection outcomes do not use this cancellation path.

This prevents an already rejected parent task from leaving participants with non-actionable pending collaboration evidence.

## 3. Trusted approval principal

Commit `433b3dbecaecf285643345b4a62e164c20676afc` introduced `ApprovalPrincipal` as the server-authenticated approval identity. It contains:

- `tenantId`;
- `operatorId`;
- bounded authority set;
- account status;
- optional session expiry.

Tenant and operator identifiers are format constrained. Authorities are deduplicated, format constrained and limited to 64 values. A disabled principal and an expired session are rejected before controller execution.

`ApprovalPrincipal` is an authentication model. It does not infer business-task participation from a management role, and it does not infer management capability from task participation.

## 4. Server-owned request context

`ApprovalIdentityContextFilter` applies to `/api/approval/**` and establishes the values consumed by the existing controller and application command construction:

- `X-Tenant-Id` is replaced with the authenticated principal tenant;
- `X-Operator-Id` is replaced with the authenticated principal operator;
- the configured trusted permission header is removed from the wrapped request;
- `getUserPrincipal`, `getRemoteUser` and `isUserInRole` are backed by `ApprovalPrincipal`;
- `X-Request-Id` and `X-Trace-Id` are validated and placed on the wrapped request and response;
- request and trace identifiers are placed in MDC for structured logging and removed in `finally`;
- absent correlation identifiers are generated by the server.

The accepted correlation format is `[A-Za-z0-9][A-Za-z0-9._:@-]{0,127}`. Control characters, whitespace, overlength identifiers and other characters are rejected. An invalid client request identifier is never echoed as the failure correlation identifier; a safe server-generated identifier is returned instead.

Because the existing controllers create `RequestContext` from request headers, the wrapped server-owned values continue through idempotency, audit, notification and Outbox code without trusting the original browser or mobile headers.

## 5. Authentication modes and fail-closed profiles

Base configuration uses:

```yaml
approval:
  security:
    identity:
      mode: ${APPROVAL_IDENTITY_MODE:principal}
```

`principal` mode requires an upstream trusted authentication integration to place an `ApprovalPrincipal` on the request. The standalone application therefore returns an authentication error rather than silently inventing an identity when no integration is installed.

`local-headers` mode is only allowed when the explicit `local` or `test` profile is active. Startup fails if that mode is selected in any other profile. `application-local.yml` selects `local-headers` for local development while keeping management permission enforcement enabled.

The local mode is an explicit development/test adapter. It parses bounded tenant, operator and permission headers into an `ApprovalPrincipal`; after that conversion the original trusted-permission header is removed and downstream authorization uses the principal.

The base configuration does not activate the local profile and does not default to local identity.

## 6. Tenant isolation and stable failures

The filter uses the following stable failures:

| Code | HTTP status | Boundary |
| --- | ---: | --- |
| `APPROVAL_AUTHENTICATION_REQUIRED` | 401 | no trusted approval principal or no local test identity |
| `APPROVAL_TENANT_CONTEXT_REQUIRED` | 401 | explicit local/test identity has no tenant |
| `APPROVAL_TENANT_CONTEXT_MISMATCH` | 404 | supplied tenant claim differs from authenticated tenant |
| `APPROVAL_PRINCIPAL_DISABLED` | 403 | account is disabled |
| `APPROVAL_SESSION_EXPIRED` | 401 | authenticated session is expired |
| `APPROVAL_REQUEST_ID_INVALID` | 400 | request identifier is malformed |
| `APPROVAL_TRACE_ID_INVALID` | 400 | trace identifier is malformed |

A cross-tenant client claim returns the same not-found style message used to avoid disclosing whether a resource exists in another tenant. A forged operator header has no effect. A forged trusted permission header has no effect in principal mode and is removed before management authorization.

## 7. Management and participant policy separation

Authentication is shared, but authorization paths remain separate:

- `/api/approval/management/**` continues to require an explicit `@ApprovalManagementPermission` capability and is fail-closed when a management handler omits the declaration;
- participant endpoints continue to rely on platform-owned task, initiator, delegation, handover, collaboration and visibility evidence;
- browser route authority and mobile UI state are not used as a server security boundary;
- an authenticated management principal does not automatically become a task assignee or collaboration participant;
- an authenticated participant does not automatically receive management capability.

Enterprise responsibility-source and resource-scope resolution is intentionally handled by the following M4-B stage rather than weakening the M4-A authentication boundary.

## 11. Accepted M4-A invariants

M4-A is accepted with these invariants:

- tenant and operator are derived from a server-authenticated approval principal in base configuration;
- browser and mobile tenant/operator headers are not trusted authority;
- a browser or mobile trusted-permission header cannot create management authority;
- local/test identity injection is explicit and cannot activate in a production profile;
- disabled and expired principals fail closed;
- cross-tenant claims do not reveal resource existence;
- request and trace identifiers are bounded, sanitized by validation, generated when absent and propagated to downstream platform evidence;
- management capability checks and participant business rights remain separate;
- the production source tree has a permanent check against Flowable internal table dependencies;
- the M3 baseline and client build boundaries remain green.
